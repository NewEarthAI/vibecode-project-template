# Anti-Vibe-Coded — The NEVER List

> **What "vibe-coded" means**: interfaces that look like they were generated by an AI reaching for the most obvious Tailwind defaults. Every vibe-coded dashboard looks the same: rounded-2xl cards, purple gradients, emoji, pastel status pills, glassmorphism, Inter font, colored shadows, one card per color. It is the new "Bootstrap default" — recognizable at a glance and immediately signals "built fast, built cheap".
>
> This skill exists to make the agency interfaces look like they were built by a senior developer in 2019, before AI code assistants existed and every tutorial looked the same. Pre-AI era interfaces signal craftsmanship through *restraint* — tighter radii, hairline borders, precise shadows, monochrome bases, earned color.

---

## Hard NEVER Rules (Audit-Enforced)

These are caught by `scripts/audit-forbidden-patterns.sh`. A single violation fails the audit.

### 1. NEVER use `rounded-2xl` or `rounded-3xl`

```tsx
// ✗ REJECTED
<Card className="rounded-2xl">...</Card>
<Card className="rounded-3xl">...</Card>

// ✓ CORRECT
<Card className="rounded-xl">...</Card>  // 12px max
<Card className="rounded-lg">...</Card>  // 8px, standard
```

**Why**: `rounded-2xl` (16px) is the "chunky iOS consumer app" look. It screams *cute*, not *premium*. Every AI-generated dashboard reaches for it by default because it's "friendly". Premium industrial software uses tighter radii (Stripe, Linear, Vercel, Notion all cap at 8-12px on cards).

**Exception**: `rounded-full` is allowed for pills, avatars, circular badges only.

---

### 2. NEVER use `backdrop-blur-*` on content surfaces

```tsx
// ✗ REJECTED — glassmorphism cliche
<div className="backdrop-blur-md bg-white/30 border border-white/20">...</div>
<Card className="backdrop-blur-lg bg-white/20">...</Card>

// ✓ CORRECT — solid surface with two-layer shadow
<Card className="bg-[var(--ne-bg-base)] shadow-[var(--ne-shadow-card)] border border-[var(--ne-border-hairline)]">
  ...
</Card>
```

**Why**: glassmorphism was cool for 18 months in 2022-2023 and is now instantly dated. It tanks performance on large surfaces (browsers have to composite blurred layers every frame), creates unpredictable text contrast against dynamic backgrounds (accessibility failure), and has the same "every SaaS landing page" feel as purple gradients.

**Exception**: the guided-tour skill owns `backdrop-filter` on its overlay. Do not touch it, do not imitate it elsewhere.

---

### 3. NEVER use emoji in code or copy

```tsx
// ✗ REJECTED
<Button>🚀 Launch</Button>
<h1>Dashboard 📊</h1>
<Badge>✨ New</Badge>
// commit: "feat: add tour button 🎉"

// ✓ CORRECT
<Button>Launch</Button>
<h1>Dashboard</h1>
<Badge variant="neutral">New</Badge>
// commit: "feat: add tour button"
```

**Applies to**:
- UI copy (button text, headings, labels, empty states)
- Component children (no decorative emoji in JSX)
- Technical documentation inside the repo
- Commit messages
- PR descriptions
- Code comments

**Why**: emoji in professional software reads as consumer-app amateur. Enterprise users perceive it as "unserious". Icons from [lucide-react](https://lucide.dev) or a licensed icon set convey the same information with precision and consistent weight.

**Exception**: user-generated content (messages from drivers, chat transcripts) may contain emoji — that is data, not UI copy. The UI *displays* it, it does not *produce* it.

---

### 4. NEVER use translucent white backgrounds (`bg-white/X`)

```tsx
// ✗ REJECTED
<div className="bg-white/30">...</div>
<div className="bg-white/50 backdrop-blur-sm">...</div>

// ✓ CORRECT
<div className="bg-[var(--ne-bg-base)]">...</div>  // #FBFBFA
```

**Why**: `bg-white/30` is the glassmorphism gateway drug. It only looks intentional over a dynamic image/gradient background, and even then it's dated. Solid neutral surfaces with proper shadows always win.

---

### 5. NEVER use `Inter`, `Roboto`, `Arial`, or `system-ui` as the primary typeface

```css
/* ✗ REJECTED */
body { font-family: Inter, system-ui, sans-serif; }
body { font-family: Roboto, Arial, sans-serif; }

/* ✓ CORRECT */
body { font-family: 'DM Sans', system-ui, -apple-system, sans-serif; }
```

**Why**: Inter is the Tailwind default. Roboto is the Material Design default. Arial is the 1990s fallback. All three mean "we didn't choose a font". DM Sans has personality (slightly geometric, warm counters, great at small sizes), is free, and is used by very few AI-generated dashboards — meaning it immediately reads as *chosen* rather than *default*.

**Exception**: client-specific licensed typefaces (corporate brand fonts like Söhne, GT America, Söhne Mono) are encouraged. System fonts are not.

---

### 6. NEVER use colored shadows

```tsx
// ✗ REJECTED
<div className="shadow-xl shadow-cyan-500/50">...</div>
<div className="shadow-lg shadow-purple-500/30">...</div>

// ✓ CORRECT
<div className="shadow-[var(--ne-shadow-card)]">...</div>
```

**Why**: colored shadows / glows read as "gaming website", "crypto landing page", or "TikTok filter". Pure grayscale depth is the premium move. Stripe, Linear, Apple — none use colored shadows on content surfaces.

---

### 7. NEVER use Tailwind's `shadow-2xl` on cards

```tsx
// ✗ REJECTED — too blurry, consumer
<Card className="shadow-2xl">...</Card>

// ✓ CORRECT — the signature two-layer shadow
<Card className="shadow-[var(--ne-shadow-card)] hover:shadow-[var(--ne-shadow-card-hover)]">...</Card>
```

**Why**: `shadow-2xl` is `0 25px 50px -12px rgba(0,0,0,0.25)` — a single huge blur. It looks like a card floating on a pillow. Real card elevation is two subtle layers (close contact shadow + distant ambient shadow).

## Soft NEVER Rules (Review-Enforced — Cannot Be Auto-Audited)

These require human judgment during review.

### 8. NEVER put multiple KPIs competing for attention with different colors

```tsx
// ✗ REJECTED — 4 colors fighting for attention
<Card className="border-blue-500 text-blue-900">Total Loads</Card>
<Card className="border-green-500 text-green-900">Avg Loads</Card>
<Card className="border-amber-500 text-amber-900">Top Performer</Card>
<Card className="border-red-500 text-red-900">Revenue</Card>
```

**Why**: if everything is urgent, nothing is. KPIs should be neutral cards with state-dependent color *inside* (e.g., variance arrows). The cards themselves are equal siblings.

---

### 9. NEVER use decorative icons inside status badges

```tsx
// ✗ REJECTED
<Badge variant="success">✓ Approved</Badge>
<Badge variant="warning">⚠ Pending</Badge>
<Badge variant="critical">✗ Rejected</Badge>

// ✓ CORRECT (text-only badges)
<Badge variant="success">Approved</Badge>
<Badge variant="warning">Pending</Badge>
<Badge variant="critical">Rejected</Badge>
```

**Why**: the color of the badge already conveys state. Adding an icon is redundant. Redundant visual language dilutes the design.

**Exception**: if the badge has no color (neutral variant) and needs to convey a state distinction, a single precise lucide-react icon at 12-14px is acceptable — but only because it is the *only* state signal.

---

### 10. NEVER use large illustrations or 3D renders as section dividers

```tsx
// ✗ REJECTED — illustration clutter
<div>
  <img src="/illustrations/trucks-driving.svg" className="h-64 w-full" />
  <h2>Fleet Performance</h2>
</div>

// ✓ CORRECT
<section>
  <h2 className="text-2xl font-semibold mb-4">Fleet Performance</h2>
  {/* content */}
</section>
```

**Why**: illustrations are for marketing pages, not operational dashboards. Enterprise users perceive them as padding. Space is created by *typography and whitespace*, not decoration.

---

### 11. NEVER use gradient backgrounds on content surfaces

```tsx
// ✗ REJECTED
<Card className="bg-gradient-to-br from-purple-500 to-pink-500 text-white">...</Card>
<div className="bg-gradient-to-r from-blue-50 to-cyan-50">...</div>

// ✓ CORRECT
<Card className="bg-[var(--ne-bg-base)]">...</Card>
```

**Why**: gradient backgrounds are the single strongest "AI-generated" tell. Every Claude/ChatGPT tutorial defaults to them. Flat neutral surfaces + two-layer shadows carry the same visual hierarchy with 1/10 the cliche.

**Exception**: the silver gradient border (Mode A/B/D) is the ONE approved gradient in the system. Nowhere else.

---

### 12. NEVER use animated loaders that feel consumer (`animate-pulse` on content, sparkle effects, shimmer gradients)

```tsx
// ✗ REJECTED
<div className="animate-pulse bg-gradient-to-r from-pink-200 via-purple-200 to-indigo-200">...</div>

// ✓ CORRECT — skeleton with flat neutral pulse
<div className="animate-pulse bg-[var(--ne-bg-muted)] h-4 w-32 rounded"></div>
```

**Why**: loaders are interruptions. They should be as invisible as possible. Animated gradients draw attention to the waiting, which is exactly backwards.

---

### 13. NEVER let buttons sprawl — one primary CTA per view

```tsx
// ✗ REJECTED — three equally-loud CTAs
<div className="flex gap-2">
  <Button variant="primary">Save</Button>
  <Button variant="primary">Apply</Button>
  <Button variant="primary">Publish</Button>
</div>

// ✓ CORRECT — hierarchy
<div className="flex gap-2">
  <Button variant="primary">Save</Button>
  <Button variant="secondary">Apply</Button>
  <Button variant="ghost">Publish</Button>
</div>
```

**Why**: every screen has one most-important action. Surfacing three primary buttons splits the user's decision. The primary color is a precious resource.

---

### 14. NEVER use cutesy empty states with mascot illustrations

```tsx
// ✗ REJECTED
<div className="text-center py-12">
  <img src="/empty-states/sad-robot.svg" className="mx-auto h-32" />
  <h3>Nothing here yet! 😔</h3>
  <p>Try adding some data to get started ✨</p>
</div>

// ✓ CORRECT
<div className="text-center py-12">
  <p className="text-sm text-[var(--ne-fg-tertiary)]">No data available.</p>
  <Button variant="ghost" className="mt-4">Add first entry</Button>
</div>
```

**Why**: mascots and friendly copy belong in consumer apps (Slack, Duolingo, Notion's onboarding). Enterprise users perceive them as patronizing. Quiet, clear, actionable empty states respect the user's time.

---

### 15. NEVER use `capitalize` CSS to fake sentence case

```tsx
// ✗ REJECTED
<h2 className="capitalize">fleet operations dashboard</h2>

// ✓ CORRECT — write it correctly
<h2>Fleet Operations Dashboard</h2>
```

**Why**: `capitalize` capitalizes *every word*, producing "Fleet Operations Dashboard" — which is Title Case, not sentence case. It also fails on proper nouns and acronyms. Write copy correctly in the source.

---

### 16. NEVER use `text-shadow` for anything

```css
/* ✗ REJECTED */
h1 { text-shadow: 2px 2px 4px rgba(0,0,0,0.3); }
```

**Why**: text-shadow is the fastest way to look like a 2005 PowerPoint slide. If text needs more emphasis, use weight, size, or color — never shadow.

## The Copy NEVER List

UI copy quality is part of the design. These rules are enforced by review.

| Wrong | Why | Right |
|-------|-----|-------|
| "Oops! Something went wrong 😢" | Consumer-apologetic | "Request failed. Retry or contact support." |
| "Awesome! Your data was saved ✨" | Infantilizing | "Saved." |
| "Let's get started!" | Fake-cheerful | "Set up your account" |
| "Loading your amazing dashboard..." | Self-congratulation | "Loading..." |
| "Nothing to see here!" | Cute-evasive | "No results." |
| "Oopsie daisy!" | Never | Error: "Unable to connect" |
| "You're all caught up! 🎉" | Cute | "No new items." |
| "Pro tip: ..." | Condescending | "Tip: ..." or just state it |

**Rule**: every copy line must pass the "would a senior enterprise buyer in a suit find this acceptable" test. If it sounds like a Duolingo owl, rewrite.

## The Icon NEVER List

- Never mix icon families (lucide + heroicons + tabler in the same project)
- Never use emoji as icons
- Never use icons at multiple sizes in the same row (pick one: 14px, 16px, 20px per context)
- Never use colored icons (unless the color conveys semantic state, and only then)
- Never use filled icons next to outline icons — pick one style per context
- Never use decorative icons that don't reinforce the label

**Approved icon library**: [lucide-react](https://lucide.dev) (already included in most shadcn-ui projects, unified visual weight, free, actively maintained).

## Quick Self-Audit Questions

Before considering any component "done", ask:

```
1. Is there any decorative color? (Remove it.)
2. Is there any decorative icon? (Remove it.)
3. Is there any emoji? (Remove it.)
4. Is the radius rounded-xl or smaller? (If not, fix it.)
5. Is there any backdrop-blur? (Remove it.)
6. Is the shadow grayscale and two-layer? (If not, fix it.)
7. Is the typography DM Sans + JetBrains Mono? (If not, fix it.)
8. Does the copy sound like a senior dev wrote it, not a chatbot? (If not, rewrite.)
9. Would I be embarrassed to show this to a senior designer at Linear or Stripe? (If yes, fix it.)
10. Does it pass all three audit scripts? (If no, fix it.)
```

## Client Brand Override Protocol

> **Critical clarification**: The Six Hard Rules and the soft NEVER list are **agency-level defaults**, not universal absolutes. When a client's documented brand guidelines legitimately conflict with a rule, the client brand wins — but the exception must be explicitly documented, not silently applied.

### When Overrides Are Allowed

A rule override is legitimate when ALL of the following are true:

1. **The client has a documented brand guideline** — an existing brand book, style guide, design system, or written brand standards document. Not a verbal preference. Not "the client likes rounded corners". A real document.
2. **The agency owner has reviewed and approved** — you (the decision-maker) have explicitly decided to honor the client brand over the the agency default.
3. **The override is scoped to the client** — it applies only in that client's project directory, never to the template, never to the audit scripts, never propagated to other clients.
4. **The override is documented in writing** — a per-client file at `clients/{slug}/design-overrides.md` that captures what was overridden, why, and when.

### When Overrides Are NOT Allowed

- **"The developer just wanted to"** — no documentation, no approval
- **"The client mentioned it casually"** — no written brand guideline
- **"Everyone does it this way"** — fashion ≠ brand requirement
- **"The rule is annoying"** — that is a feature, not a bug
- **Applied to multiple clients as a shortcut** — that's a rule change, not an override

### Override File Structure

Create `clients/{client_slug}/design-overrides.md` with this exact structure:

```markdown
# Design Overrides — {Client Name}

**Client**: {Full Legal Client Name}
**Project start**: {YYYY-MM-DD}
**Reviewed by**: {agency owner name}
**Last updated**: {YYYY-MM-DD}

## Why This File Exists

The the agency house style (see `.claude/skills/ui-design-system/`) defines
default rules that apply to every project. This file documents the specific
cases where {Client Name}'s brand guidelines require deviations from those
defaults. Every entry below has been reviewed and approved.

## Active Overrides

### Override 1: {Short title}
- **the agency default**: {quote the rule being overridden}
- **Client requirement**: {quote from client brand guideline — include document name and section}
- **Approved scope**: {where does this apply — specific components, all components, specific pages?}
- **Rationale**: {one sentence why the client brand wins here}
- **Expiration**: {date or "indefinite" — forces review}

### Override 2: {...}
```

### Example — Hypothetical "Velvet Cosmetics" Client

```markdown
# Design Overrides — Velvet Cosmetics

**Client**: Velvet Cosmetics (Pty) Ltd
**Project start**: 2026-07-15
**Reviewed by**: {agency owner}
**Last updated**: 2026-07-15

## Why This File Exists
the agency house style caps border radius at `rounded-xl` (12px). Velvet's brand
guidelines specifically require `rounded-2xl` (16px) on all CTA buttons and
hero cards as a brand signature element.

## Active Overrides

### Override 1: rounded-2xl allowed on CTAs and hero cards
- **the agency default**: "No rounded-2xl or rounded-3xl — cap at rounded-xl (12px)"
  (from `anti-vibe-coded.md` Hard Rule 1)
- **Client requirement**: Velvet Cosmetics Brand Book v3.2, Section 4.1
  "Corner Radius": "All interactive elements use 16px radius as a brand signature"
- **Approved scope**: Only Button (`variant="primary"`) and Card (`premium` variant).
  Everything else still caps at rounded-xl.
- **Rationale**: Brand signature element, documented in client's brand book,
  client has seen the agency's default and explicitly requested the override.
- **Expiration**: Indefinite (brand signature)
```

### Audit Script Behavior

The three audit scripts (`audit-forbidden-patterns.sh`, `audit-colors.sh`,
`audit-hover-consistency.sh`) will flag violations based on the default the agency
rules. When a `clients/{slug}/design-overrides.md` file exists, you can
suppress known-override violations by passing the overrides file as an exclusion:

```bash
bash audit-forbidden-patterns.sh . --exclude-overrides clients/velvet-cosmetics/design-overrides.md
```

*(Note: the `--exclude-overrides` flag is a planned future enhancement.
Until implemented, manually review audit output against the overrides file.)*

### The Commitment Anchor

The the agency aesthetic is **agency policy forever** — committed 2026-04-10 in
council session `council/sessions/2026-04-10-ui-design-system-packaging.md`.
This override protocol does NOT weaken that commitment. It formalizes the
distinction between "agency default" (what every the agency project looks like
by default) and "legitimate client brand exception" (documented, approved,
scoped, dated). Without a documented brand guideline AND agency owner approval,
the default rules apply absolutely.

### Anti-Patterns Of Overrides

| Wrong | Right |
|-------|-------|
| Override without a written client brand guideline | Require a real brand book / style guide reference |
| Override approved verbally, undocumented | File exists with date, approver, rationale |
| One override file copied across clients | One override file PER client, never shared |
| Override scope "all components" without specifics | Specific component names, specific pages |
| Override with no expiration date | Either date-bounded OR explicit "indefinite (brand signature)" |
| Override that changes the template defaults | Overrides live in `clients/`, never in `.claude/skills/` |

## L2 Compositions — Net-New Bans from `design-taste-frontend`

The L2 anti-slop overlay (`.claude/skills/design-taste-frontend/SKILL.md`, MIT, Leonxlnx 2026) layers atop these house rules. Most of its bans overlap (purple gradients, emoji, glassmorphism, Inter default, gradient backgrounds) — the L2 framing reinforces them. The bans below are **net-new** items the L2 surfaces that house rules did not previously codify. Added per Design Suite v2 amendment A8 (council 2026-05-13).

These compose with the existing 16 NEVER rules — house rules are absolute, some L2 bans are dial-driven (fire above `DESIGN_VARIANCE`, `MOTION_INTENSITY`, or `VISUAL_DENSITY` thresholds defined in the L2 skill).

### 17. NEVER spam `z-50` / `z-10` without a systemic layer context

```tsx
// REJECTED
<div className="z-50">Tooltip</div>
<div className="z-50">Card hover state</div>
<div className="z-50">Random overlay</div>

// CORRECT — reserve z-indexes for systemic stack contexts
<div className="z-50">  {/* Modal */}
<div className="z-40">  {/* Sticky nav */}
<div className="z-30">  {/* Tooltip */}
```

**Why**: arbitrary z-index sprinkling creates an invisible stacking war. The fix is always retroactive and painful. Reserve z-indexes for documented systemic layers (modals, sticky navs, tooltips, drawers) — never as a quick fix for "this element should be on top".

---

### 18. NEVER animate `top` / `left` / `width` / `height`

```tsx
// REJECTED — triggers layout recalc + paint every frame
<motion.div animate={{ left: 100, width: 200 }} />

// CORRECT — GPU-accelerated, no layout cost
<motion.div animate={{ x: 100, scale: 1.2 }} />
```

**Why**: animating layout properties forces the browser to recalculate layout (reflow) and repaint on every frame. On mobile this collapses to <30fps. Always animate `transform` and `opacity` only — both are GPU-composited and have zero layout cost.

---

### 19. NEVER use `window.addEventListener('scroll', ...)` for scroll-driven animation

```tsx
// REJECTED — fires every scroll event, JS thread blocked
useEffect(() => {
  window.addEventListener('scroll', handler);
  return () => window.removeEventListener('scroll', handler);
}, []);

// CORRECT — Framer Motion uses requestAnimationFrame internally
import { useScroll, useTransform } from 'framer-motion';
const { scrollYProgress } = useScroll();
const opacity = useTransform(scrollYProgress, [0, 1], [0, 1]);
```

**Why**: raw scroll listeners fire many times per scroll event and run on the main thread. Framer Motion's `useScroll` and `useTransform` use `requestAnimationFrame` and `MotionValue` (outside React render cycle) — silky on every device.

---

### 20. NEVER place grain/noise filters on scrolling containers

```tsx
// REJECTED — repaints the noise filter every scroll frame
<div className="overflow-scroll filter contrast-150 [filter:url(#noise)]">
  ...scrolling content...
</div>

// CORRECT — fixed overlay, pointer-events:none, GPU-composited
<div className="fixed inset-0 z-50 pointer-events-none [filter:url(#noise)]" />
<div className="overflow-scroll">...scrolling content...</div>
```

**Why**: filters on scrolling containers force GPU repaints on every scroll frame. Mobile devices melt. Always apply grain/noise as a fixed, full-viewport, pointer-events:none overlay on a pseudo-element — paints once, composites forever.

---

### 21. NEVER fluctuate between warm and cool grays within one project

```tsx
// REJECTED — one card uses zinc, another slate, another neutral
<Card className="bg-zinc-50">...</Card>
<Card className="bg-slate-50">...</Card>
<Card className="bg-neutral-50">...</Card>

// CORRECT — pick one gray family, use it everywhere
<Card className="bg-zinc-50">...</Card>
<Card className="bg-zinc-50">...</Card>
```

**Why**: warm grays (zinc, stone) read totally differently from cool grays (slate, gray) under both light and dark themes. Mixing them creates a subtle "two different designers worked on this" feel. Pick one family per project and stick to it. the agency default is warm cream `#FBFBFA` + cool silver — that combination is the signature; do not introduce a third temperature.

---

### 22. NEVER box high-density data in generic `<Card>` wrappers (when `VISUAL_DENSITY > 7`)

```tsx
// REJECTED — high-density data inside boxed cards reads as suffocating
<div className="grid grid-cols-4 gap-2">
  {metrics.map(m => (
    <Card key={m.id}><CardContent>{m.value}</CardContent></Card>
  ))}
</div>

// CORRECT — use logic-grouping primitives, let data breathe
<div className="grid grid-cols-4 gap-x-6 gap-y-4 divide-y divide-zinc-200">
  {metrics.map(m => (
    <div key={m.id} className="py-3">{m.value}</div>
  ))}
</div>
```

**Why**: at high data density (pilot-cockpit, financial dashboards, ops monitoring), generic card wrappers each draw a box around every datum. The result is visual fatigue. Replace with logic-grouping primitives (`border-t`, `divide-y`, negative space) and let the data breathe. Cards are for *elevation* (a card that *clicks*), not for *grouping* (an arbitrary visual fence).

## L2 Composition Rules (How These Layer)

1. **House rules (#1-#16) are absolute** — they apply regardless of `DESIGN_VARIANCE` / `MOTION_INTENSITY` / `VISUAL_DENSITY` dial values.
2. **L2 net-new bans (#17-#22) compose** — most apply absolutely (z-index, transform-only animation, scroll listeners, filter placement, gray-family consistency). The card-overuse ban (#22) is dial-driven (`VISUAL_DENSITY > 7`).
3. **Order of application** when generating output: L1 register (`PRODUCT.md` + `DESIGN.md`) → L2 dial baseline (`design-taste-frontend`) → L3 house tokens (this skill).
4. **Conflicts**: when L2 and house rules overlap, the stricter rule wins. House rules' tighter framing (e.g. cap at `rounded-xl` not `rounded-lg`) supersedes L2's looser equivalents.
